v1.9.0 · procedural memory

Padrões com quality score.

Brains são a memória procedural dos agentes — recipes específicas com q de 1 a 5, anti-patterns marcados (v: AVOID), carregados sob demanda por tag matching. Diferente de rules (boas práticas amplas), brains armazenam know-how validado pelo uso. Hoje: 3 brains, 25 nodes.

Por que separar de rules: rules são princípios amplos ("escreva tests"). Brains são recipes específicas ("para evitar shell injection, use spawn(cmd, [args]) ao invés de execSync com template literal"). Brains também trazem variante v: AVOID pra anti-patterns conhecidos — algo que rules não capturam bem. Agentes carregam só nodes que fazem match com as tags da task atual, com filtro q ≥ 4 por default — recipes ruins não vazam pro contexto.

Anatomia · um node

Estrutura de uma recipe.

Cada node é um JSON com 6 campos canônicos. Tags fazem discovery; quality score gate de qualidade; variante define se é PREFER ou AVOID.

id
Slug curto único dentro do brain. Ex: spawn-with-array-args, shell-injection-defense.
tags
Array de termos pra discovery via --tags=a,b. Múltiplas tags por node — match parcial conta.
q
Quality score 1-5. Filtros default usam ≥ 4. Honestidade do agente que adiciona — não auto-inflar.
v
Variante: PREFER (padrão recomendado) ou AVOID (anti-pattern conhecido). Nodes AVOID têm campo not com o que NÃO fazer.
desc
Uma linha. Recipe acionável. Não explica o porquê — explica o quê fazer (ou evitar).
see
Cross-refs pra outros nodes relacionados. Permite agente seguir thread de conhecimento.

Loading flow · 4 passos

Como o agente consulta.

Tiered loading: index sempre primeiro (~2KB), brain files completos só sob demanda quando tags casam.

01

Load index

Agente carrega _index.json (~2KB) no preflight. Lista todos brains com tags + counts.

02

Tag match

Para task atual, identifica tags relevantes (ex: security + shell). Match contra index.

03

Load matched

Só brains com tag match são carregados. dev/patterns para task de security; outros ficam fora.

04

Filter by q

Default q ≥ 4. Recipes baixa qualidade não vazam pro contexto. v: AVOID aplicado como restrição.

brain:query

CLI · 1 comando

Buscar padrões via brain:query.

Brains não têm CRUD CLI — agentes editam diretamente os .brain.json files. Query é a única operação read-only deterministic.

aioson brain:query
Buscar por tag
Flags: --tags=a,b (csv) · --agent=<name> (filtro adicional) · --min-quality=4 (default) · --format=compact|json|ids. Sem chamada LLM — pure index lookup + JSON filter.
Tutorial · criar próprio brain

5 passos pra ter seu brain.

Brains podem ser específicos de um projeto. Útil quando você acumula padrões próprios (ex: domain-specific patterns).

1

Crie o diretório do brain

Em .aioson/brains/<categoria>/. Categoria pode ser por agente (dev/), por domínio (marketing/) ou por projeto.

2

Escreva o brain file

Arquivo <slug>.brain.json com schema: { id, version, agents, nodes: [...] }. Cada node com id, tags, q, v, desc, see.

3

Adicione ao index

Edite .aioson/brains/_index.json acrescentando entrada com id, agents, tags, desc, path, nodes count, updated.

4

Teste com brain:query

Verifique se as tags retornam os nodes corretos. aioson brain:query . --tags=<suas-tags> --format=compact.

5

Evolução natural

Conforme @dev/@sheldon (ou seus agentes customizados) identificam novos padrões, eles acrescentam nodes ao brain — com q honesto + see refs. updated e nodes count em _index.json também atualizam.
