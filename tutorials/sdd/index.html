SDD em uma frase: ao invés do agente "tentar implementar direto" a partir de um prompt, ele segue um pipeline determinístico — primeiro entende o produto (@product), depois mapeia entidades (@analyst), define decisões estruturais (@architect), implementa (@dev), e valida com QA (@qa). Classificação · MICRO / SMALL / MEDIUM

O processo escala com o tamanho.

Nem toda feature precisa do pipeline completo. aioson classify mapeia a feature em um dos 3 tiers; cada um ativa uma cadeia diferente de agentes e gates.

Tier Quando Chain de agentes Duração típica
MICRO Sem entidades novas. Fix, refactor pequeno, alias, nova flag de CLI. --no-distill opt-in. @product → @dev → @qa 1-2 sessões
SMALL Entidade nova ou refactor médio. Webhooks, integrações de 1 hop, novas views. @product → @analyst → @dev → @qa 3-5 sessões
MEDIUM Multi-entidade, multi-hop, paralelismo, harness contract. Living Memory, Squads, novos agentes. @product → @sheldon → @analyst → @architect → @dev → @qa → @validator 1-3 semanas

Tutorial · sua primeira feature

Do zero ao Gate D PASS.

5 passos pra entregar uma feature SMALL completa. Cada passo aciona um agente específico — não improvisa, não pula etapa.

1

Ative /neo pra orientar

Neo lê o estado do projeto (pulse + Living Memory) e rotea pro próximo agente. Em projetos novos, vai direcionar pra /setup ou /product.

/neo
$ /neo project: meu-app · stage: ready for next feature → recomendado: /product (sem feature ativa)
2

Escreva o PRD com /product

@product conduz uma conversa estruturada — vision, problem, users, MVP must-have/should-have, out-of-scope, métricas. Saída: prd-{slug}.md + entrada em features.md.

Em profile creator, faz 1 pergunta por turno via AskUserQuestion com opções. Em profile developer, batcheia até 5 perguntas.

3

Map de entidades com /analyst

SMALL+ entra aqui. @analyst extrai entidades, business rules, acceptance criteria, edge cases. Verifica integridade vs discovery.md. Gate A ✓ → Gate B em construção.

4

Implemente com /dev

@dev lê dev-state.md (auto-resume sem cold start), carrega só os arquivos do context_package, executa em slices validadas. Roda memory:status pra Living Memory bootstrap gate.

/dev cold start
$ /dev resume: feature=checkout · phase=2 · next='Implement payment webhook' context_package: project.context.md, spec-checkout.md, implementation-plan-checkout.md memory:status: bootstrap 4/4 · brains 3 indexed
5

Valide e feche com /qa

@qa faz risk-first review, testa critical paths, gera test plan, decide PASS/FAIL. Em MEDIUM com harness-contract, recomenda /validator antes do close. Após PASS, feature:close dispara distillation automática.

v1.9.0 · 5 phases shipped
Living Memory

Antes do v1.9.0, agentes começavam toda sessão lendo 4-6 arquivos de contexto pra entender onde a coisa parou. Living Memory cortou isso: 4 arquivos de bootstrap auto-sincronizados + reflect engine que classifica diffs pós-agent:done + FTS5 search sobre rules/learnings.

Os 4 arquivos de bootstrap

  • what-is.md — identidade do sistema e usuários
  • what-it-does.md — capabilities ativas e improvement focus
  • how-it-works.md — fluxos e contratos internos
  • current-state.md — o que o sistema já tem (append-only)

Reflect engine

Depois de agent:done ou workflow:next --complete, o engine analisa o git diff e classifica se o impact justifica atualizar o bootstrap. Se sim, escreve um manifest em .aioson/runtime/reflect-prompt.json. O agente da próxima sessão lê o manifest, edita os bootstraps listados, e commita via memory:reflect-commit (que valida frontmatter, snapshot hash, allowed paths).

FTS5 search

memory:search "harness gate" retorna matches em content_items + learnings com ranking BM25. Phrase-query sanitization default + 500-char cap. Saída JSON pronta pra dashboard ou pro próprio agente consumir.

Doctor checks novos

bootstrap_coverage · rule_staleness · learning_orphans · distillation_lag · permissions_in_sync · jargon_leak_detection. Todos warning-level, não bloqueiam, mas surfaceiam gaps. --fix auto-corrige permissions e gaps estruturais.
v1.9.0 · procedural memory
Brains

Brains são memória procedural dos agentes — padrões que ele recorre ao implementar/decidir. Diferente de rules (boas práticas amplas), brains armazenam recipes específicas com quality score. Agente só carrega o que faz match com tags da task atual, com q ≥ 4 por default.

Brains atuais (3 · 25 nodes)

  • site-forge/visual-patterns — 14 nodes · CSS, animations, cloning
  • dev/patterns — 5 nodes · security, shell, git, editing
  • sheldon/architecture-decisions — 6 nodes · sandbox, parity, sizing

Loading lazy via tags

aioson brain:query . --tags=security,shell --min-quality=4 --format=compact retorna só os nodes que casam. Saída: compact (linha por node), json (estruturado), ids (só ids).

Estrutura de um node

  • id · slug curto
  • tags · termos pra discovery
  • q · quality score 1-5 (filtros usam ≥ 4)
  • v · variante: PREFER ou AVOID (anti-pattern)
  • desc · uma linha
  • see · cross-refs pra outros nodes

Auto-update

Quando um agente identifica um novo padrão reutilizável, ele acrescenta um node ao brain com q honesto. _index.json é atualizado com novo count + updated date. Brains evoluem com cada feature.
v1.9.0 · autonomy-protocol.json v1.1
Autonomy 3 tiers

Contrato de permissões em 3 tiers materializado pelo permissions-generator em arquivos nativos de cada harness (Claude Code, Codex, Gemini, OpenCode). Tier3 é hard-rejected mesmo se um tool listar — ações irreversíveis sempre exigem mão humana.

Tier 1 — silent

Leituras + telemetria. Não requer notificação ao usuário. Exemplos: preflight, memory:status, brain:query, context:load, memory:search.

Tier 2 — notified

Writes em memória (rules/brains/learnings). Notifica o usuário via aioson notify --level=warn antes de mutar. Exemplos: memory:archive, memory:restore, memory:reflect-commit.

Tier 3 — blocking

Irreversível. Hard-rejected pelo harness regardless of allow-list. Exige humano explícito. Exemplos: git push, npm publish, cloud:publish:*, genome:publish, skill:publish, squad:publish.

Como funciona o materializer

No install/update, src/permissions-generator.js lê autonomy-protocol.json e gera arquivos nativos:

  • .claude/settings.json
  • .codex/permissions.json
  • .gemini/permissions.toml
  • .opencode/permissions.yaml

Backups em .aioson/backups/{ts}/permissions/. Doctor permissions_in_sync detecta drift e --fix regenera.
v1.9.0 · decision-presentation skill
Profile creator/developer/team

Em project.context.md, o campo profile define como agentes interagem. Default é creator (mais seguro pra não-devs). Devs trocam manualmente pra developer.

creator (default)

  • 1 pergunta por turn via AskUserQuestion com 2-4 opções estruturadas
  • Marcador (Recomendado) obrigatório na primeira opção, com plain-language why
  • Jargão traduzido via dicionário (MICRO → "feature pequena", Gate A → "checkpoint de requisitos", etc)
  • "Pausar / quero pensar" sempre disponível como última opção
  • Sem free-form questions abertas (escape via "Other / Conte com suas palavras")

developer

  • Cadência permissiva: até 5 numbered questions por batch
  • Jargão permitido sem tradução
  • Marker (Recomendado) opcional

team

  • Cadência igual a developer no fluxo operacional
  • Emite executive summary em agent:done usando o dicionário (operadores leem jargão; stakeholders leem tradução)

Onde se aplica

Agentes com a skill decision-presentation wired: @neo, @setup, @product, @dev, @deyvin. Doctor check jargon_leak_detection detecta vazamento de termos não traduzidos em projetos creator.

Migração

Projetos legacy com profile: beginner migram automaticamente pra creator no próximo aioson update. Notify visual surfaceia a mudança.
v1.8.0+ · feature-level memory
Dossier por feature

Cada feature SMALL ou MEDIUM ganha .aioson/context/features/{slug}/dossier.md — uma memória dedicada da feature, separada da memória global do projeto. Auto-init silencioso após PRD pra classificações SMALL+.

Seções padrão do dossier

  • Why — motivação, problema, stakeholders
  • What — escopo MVP, must-have, out-of-scope
  • Code Map — paths tocados em cada slice
  • Agent Trail — pass por pass (cada agente adiciona finding)
  • Research — items de researchs/ consumidos
  • Rules linked — rules de .aioson/rules/ consultadas

Para que serve

  • Cold start barato — @dev/@deyvin leem dossier antes do PRD; entendem contexto em segundos
  • Drift detection — antes de modificar arquivo, checam se está em code_map e se mudança diverge do plan
  • Audit trail — quem decidiu o que, quando, com que razão
  • Compactação — dossier:compact resume entradas antigas mantendo decisões importantes

Auto-archive em feature:close

Quando feature:close --verdict=PASS roda, o dossier é arquivado em .aioson/context/done/{slug}/dossier/ junto com PRD, spec, requirements. Fica recuperável mas não polui o contexto ativo.
MEDIUM only · Gate D enforcement
Harness contracts

Para features MEDIUM, Gate D vira enforcement programático. @sheldon gera .aioson/plans/{slug}/harness-contract.json com binary criteria extraídas das ACs do PRD. @validator roda headless e avalia cada criterion. Circuit breaker bloqueia feature:close até passar.

Quando ativa

Só em MEDIUM (RF-05 do @sheldon). MICRO e SMALL não geram harness-contract — gates A·B·C·D continuam por inspeção semântica.

Anatomia do contract

  • criteria[] — array de checks binários (id, description, evaluator)
  • circuit_state — CLOSED (default), HALF_OPEN, OPEN
  • error_streak_limit — quantos fails consecutivos abrem o circuito
  • ready_for_done_gate — flag que feature:close consulta

Fluxo

  1. @dev termina implementação → progress.status = 'waiting_validation'
  2. workflow:next auto-routes pra @validator (detour temporário)
  3. @validator avalia cada criterion, escreve last-validator-output.json
  4. harness:apply-validation consome output, atualiza progress.json
  5. Se todos passam: ready_for_done_gate = true
  6. Se falha: last_error capturado, circuit breaker incrementa
  7. feature:close --verdict=PASS verifica ready_for_done_gate antes de fechar

Bypass

feature:close --force bypassa com audit trail entry "BYPASS" registrada. --verdict=FAIL skipa o gate inteiro (QA já rejeitou).
